# 核心架构设计

## 1. 设计目标

本项目的核心架构目标有四点：

- 用统一数据模型承载 Stack / MPR / VR 三种视图
- 明确区分数据层、视图层、控制层和后台任务层
- 尽量复用切片显示能力，避免为 Stack 和 MPR 维护两套底层实现
- 让多线程加载、渲染管线和 UI 交互能够自然解耦

## 2. 总体结构

建议采用如下分层：

```text
+---------------------------------------------------------------+
| Application Shell                                             |
| MainWindow / Menu / Toolbar / WorkspaceWidget                 |
+---------------------------------------------------------------+
| Page Layer                                                    |
| StackPage / MPRPage / VRPage / SeriesSelectionDialog          |
+---------------------------------------------------------------+
| View Layer                                                    |
| SliceViewWidget / VolumeViewWidget / ThumbnailPanel           |
+---------------------------------------------------------------+
| Controller & Service Layer                                    |
| ImportController / MPRController / ThumbnailService           |
| VolumeBuilder / ViewerStateManager                            |
+---------------------------------------------------------------+
| Data Model Layer                                              |
| DicomSeries / DicomSliceInfo / VolumeData / ViewerState       |
+---------------------------------------------------------------+
| Infrastructure                                                |
| DCMTK / VTK / Qt / spdlog / QThreadPool or QtConcurrent       |
+---------------------------------------------------------------+
```

## 3. 核心数据关系

项目采用统一体数据模型：

```text
本地目录
   -> DICOM 文件扫描
   -> 按 SeriesInstanceUID 分组
   -> 用户选定一个 DicomSeries
   -> 排序与基础校验
   -> 构建 VolumeData
   -> Stack / MPR / VR 共享同一份 VolumeData
```

核心关系可以概括为：

`一个 DICOM 序列 -> 一个 VolumeData -> 多个 View（Stack / MPR / VR）`

这是第一版最重要的架构约束。它可以保证：

- 数据来源统一
- 视图行为一致
- 避免不同模式之间维护重复数据副本
- 为 MPR 与 VR 的扩展留出清晰基础

## 4. 关键对象设计

### 4.1 DicomSliceInfo

表示单个 DICOM 文件的基础信息，建议包含：

- 文件路径
- SOPInstanceUID
- InstanceNumber
- ImagePositionPatient
- ImageOrientationPatient
- PixelSpacing
- SliceThickness
- Rows / Columns
- RescaleSlope / RescaleIntercept
- 推荐窗宽 / 窗位

用途：

- 参与 series 摘要生成
- 参与排序与校验
- 为缩略图生成提供单 slice 输入

### 4.2 DicomSeries

表示用户最终可选择的一个序列，建议包含：

- SeriesInstanceUID
- SeriesDescription
- Modality
- slice 列表
- slice 数量
- 基础显示摘要信息

用途：

- 作为导入流程中的业务实体
- 作为构建 `VolumeData` 的直接输入

### 4.3 VolumeData

`VolumeData` 是三种视图共享的统一体数据对象，建议包含：

- 体素 buffer 或 `vtkImageData`
- dimensions
- spacing
- origin
- 方向信息或方向矩阵
- 标量范围
- 默认窗宽 / 窗位
- 当前 series 的基础摘要信息

用途：

- Stack View 根据当前索引显示某一张切片
- MPR View 从中提取 sagittal / coronal / axial 三个正交切面
- VR View 直接进行体渲染

### 4.4 ViewerState

建议将界面状态抽象为独立状态对象，至少包含：

- 当前 `ViewMode`：Stack / MPR / VR
- 当前 `ToolMode`
- 当前 Stack slice index
- 当前 MPR crosshair 位置
- 当前 MPR active view
- 当前窗口的 WW / WL
- 当前缩放和平移状态
- MPR sync/link 是否开启

这样可以减少状态散落在多个 widget 中，便于后期排查与扩展。

## 5. 页面与视图组件设计

### 5.1 MainWindow

职责：

- 提供菜单栏和工具栏
- 响应 `Open Folder`
- 承载 `WorkspaceWidget`
- 承载右侧缩略图栏
- 根据当前模式切换对应工具栏

### 5.2 WorkspaceWidget

职责：

- 通过 `QStackedWidget` 管理 `StackPage`、`MPRPage`、`VRPage`
- 在页面切换时维持统一的 `VolumeData` 上下文
- 向不同页面分发当前选中的序列数据

### 5.3 StackPage

职责：

- 管理单个 `SliceViewWidget`
- 响应当前 slice 变化
- 管理 Stack 专属工具，例如 Flip / Invert

### 5.4 MPRPage

职责：

- 管理三个 `SliceViewWidget`
- 维护 `active view`
- 维护 crosshair 显示与同步逻辑
- 协调三视图联动

注意：

- MPR 的特殊能力不通过重新造一套底层视图组件实现
- 仍然复用 `SliceViewWidget`
- MPR 专属逻辑由 `MPRPage` 或单独控制器负责协调

### 5.5 VRPage

职责：

- 管理单个 `VolumeViewWidget`
- 处理 preset 切换
- 处理 Reset Camera

### 5.6 SliceViewWidget

`SliceViewWidget` 是整个项目中最关键的复用组件，负责承载通用切片能力。

建议职责：

- 内部持有 `QVTKOpenGLNativeWidget`
- 管理对应的 VTK render window / renderer / image actor
- 显示当前切片或当前正交切面
- 实现 Pan / Zoom / Window-Level / Measure
- 绘制 overlay
- 以可选方式支持 crosshair

它不应知道自己当前是 Stack 还是 MPR 页面，只应知道：

- 当前数据源 `VolumeData`
- 当前显示方向或切面类型
- 当前显示索引或切面位置
- 当前交互工具状态

这样可以最大程度复用渲染和交互逻辑。

### 5.7 VolumeViewWidget

职责：

- 内部持有 `QVTKOpenGLNativeWidget`
- 封装体渲染所需 renderer、mapper、volume actor
- 响应 preset 切换
- 响应 Rotate / Pan / Zoom / Reset Camera

建议第一版优先使用 `vtkSmartVolumeMapper` 之类的自适应 mapper，以减少底层选择负担。

## 6. 导入与数据构建流程

建议流程如下：

```text
Open Folder
   -> 扫描目录中文件
   -> 过滤 DICOM 文件
   -> 读取必要 tag
   -> 按 SeriesInstanceUID 分组
   -> 生成 series 摘要列表
   -> 用户选择一个 DicomSeries
   -> 对该 series 做排序与基础一致性校验
   -> 构建 VolumeData
   -> 通知 Workspace 和 ThumbnailPanel 加载
```

基础校验建议至少覆盖：

- slice 数量是否足够
- 图像尺寸是否一致
- spacing 是否合理
- 关键元数据是否缺失
- 排序后相邻切片位置是否基本连续

实现上应避免仅依赖 `InstanceNumber` 排序。第一版可以优先参考：

- `ImagePositionPatient`
- `ImageOrientationPatient`
- 必要时再回退到 `InstanceNumber`

这会让你的设计更像真实工程，而不是样例工程。

## 7. 渲染管线设计

### 7.1 Stack 渲染

Stack View 基于统一的 `VolumeData` 显示当前 slice。建议思路：

- 输入：`VolumeData`
- 根据当前索引选择 axial 方向某一层
- 映射窗宽 / 窗位
- 交给 VTK 进行 2D 显示

Stack 的实现重点不是复杂重建，而是：

- 鼠标交互状态清晰
- 图像显示与 overlay 刷新及时
- 滚轮翻页体验稳定

### 7.2 MPR 渲染

MPR 的三个窗格都从同一份 `VolumeData` 中提取切面：

- sagittal
- coronal
- axial

核心不是“有三个窗口”，而是“它们共享同一个空间位置语义”。

建议将 crosshair 的基准状态统一保存在世界坐标或体素坐标中，然后由三个视图各自投影显示。这样更利于解释联动逻辑：

- 在任一视图滚动或点击
- 更新统一的 crosshair 位置
- 其他两个视图据此刷新切面与十字线

### 7.3 VR 渲染

VR View 直接消费 `VolumeData`，渲染策略保持简单：

- 使用基础体渲染 mapper
- 使用 3 个硬编码 preset
- 优先保证旋转、缩放、平移时的可操作性

若性能不足，可采用简单策略：

- 交互中降低采样质量
- 停止交互后恢复较高质量

这类策略在面试中很有展示价值，因为它体现的是性能取舍意识。

## 8. 多线程与 UI 解耦

第一版最值得展示的线程设计主要有两块：

### 8.1 目录扫描与 series 分组

建议放到后台线程执行，避免导入较大目录时卡死 UI。

后台线程负责：

- 遍历文件
- 读取必要 DICOM tag
- 分组并生成 series 摘要

主线程负责：

- 弹出进度或等待状态
- 展示 series 选择对话框
- 接收最终结果并更新界面

### 8.2 缩略图生成

建议采用后台任务 + 主线程回传更新的结构。

后台线程负责：

- 读取单张 slice 像素数据
- 完成基础窗宽窗位映射
- 缩放生成 `QImage` 或 `QPixmap`

主线程负责：

- 更新 Model/View
- 处理滚动、选中、点击跳转

为了保证可控性，第一版建议加入：

- 懒加载
- 邻近预取
- 简单缓存
- 任务过期丢弃机制

这样就足够体现“多线程加载与 UI 解耦”的工程思路。

## 9. 日志与可观测性

建议使用 `spdlog` 记录以下关键节点：

- 打开目录
- 扫描文件数量
- 识别出的 series 数量
- 选中的 series 信息
- `VolumeData` 构建耗时
- 缩略图任务提交与完成数量
- Stack / MPR / VR 初始化成功或失败

这些日志的价值不仅在调试，也在于面试时能够辅助说明你有工程化意识。
